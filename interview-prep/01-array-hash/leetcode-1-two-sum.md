# [LeetCode 1. 两数之和 (Two Sum)](https://leetcode.com/problems/two-sum/)

> 难度：Easy　|　标签：数组、哈希表　|　**面试出现频率 #1**

---

## 一、题目

给定一个整数数组 `nums` 和一个目标值 `target`，请在数组中找出 **和为目标值** 的那 **两个** 整数，并返回它们的数组下标。

**约束**

- `2 <= nums.length <= 10^4`
- 只会存在一个有效答案；不能使用同一个元素两次
- 返回顺序不限

**示例**

| 输入 | 输出 |
|---|---|
| `nums = [2,7,11,15], target = 9` | `[0,1]` |
| `nums = [3,2,4], target = 6` | `[1,2]` |

---

## 二、解题思路（学习重点）

### 1. 思考路径

- **暴力**：双层循环 O(n²)。是大多数人第一反应，**面试时要先讲出来**，再优化。
- **观察瓶颈**：暴力的内层循环本质是在做"查找"——给定一个数 `nums[i]`，看 `target - nums[i]` 在数组里是否出现过。**任何"查找"问题都应该联想到哈希表把 O(n) 砍到 O(1)**。
- **优化**：对每个 `nums[i]`，我们要找的是 `target - nums[i]`；如果之前出现过它，就找到了。
  → 用 **哈希表** 存"值 → 下标"，一次遍历 O(n)。
- **为什么是"一次遍历"而不是"两次（先全部入表，再扫一遍）"**：两次写法会让一个数命中自己（违反"不能使用同一元素两次"），还得额外判断下标不等；一次遍历的"先查后写"天然避开这个问题。

> **学习点 ①**：当问题形式是"**找一对/三元组满足某种和**"时，第一反应应该是 **哈希记录见过的数** 或 **排序 + 双指针**。

### 2. 一次遍历的正确性

边遍历边查询、边写入：
- 查询时，`map` 里只包含 `i` 之前的下标，**不可能查到自己**，自然满足"不能使用同一元素两次"。
- 假设答案是 `(i, j)` 且 `i < j`，遍历到 `j` 时 `nums[i]` 一定已经入表，必能命中。

### 3. 容易踩的坑

| 坑 | 处理 |
|---|---|
| 先全部 `put` 再查询 → 会命中自己 | **边查边写**，保证查询时表里不含当前元素 |
| 重复值（如 `[3,3], target=6`）丢失 | 边查边写解法天然正确：`i=0` 查不到，写入 `3→0`；`i=1` 查到 `3` 对应下标 0 |
| 用数组存值 → 还得 O(n) 查 | 用 `HashMap` 才能 O(1) 查 |

## 三、Java 题解

### 解法 A：哈希表一次遍历（推荐）

```java
class Solution {
    public int[] twoSum(int[] nums, int target) {
        Map<Integer, Integer> seen = new HashMap<>(); // value -> index
        for (int i = 0; i < nums.length; i++) {
            int need = target - nums[i];
            if (seen.containsKey(need)) {
                return new int[]{seen.get(need), i};
            }
            seen.put(nums[i], i);
        }
        return new int[0]; // 题目保证一定有解，仅满足编译
    }
}
```

**记忆口诀**：
> **"边查边写，先查后写。"** （查的是 `need`，写的是当前值）

### 解法 B：暴力（面试只用来"垫场"）

```java
for (int i = 0; i < n; i++)
    for (int j = i + 1; j < n; j++)
        if (nums[i] + nums[j] == target) return new int[]{i, j};
```

| 解法 | 时间 | 空间 |
|---|---|---|
| 哈希一次遍历 | **O(n)** | O(n) |
| 暴力 | O(n²) | O(1) |

## 五、示例验证

`nums = [2,7,11,15], target = 9`

| i | nums[i] | need | seen | 命中？ |
|---|---|---|---|---|
| 0 | 2 | 7 | `{}` | 否 → 写 `2→0` |
| 1 | 7 | 2 | `{2→0}` | **是**，返回 `[0,1]` ✅ |

## 六、复盘与延伸

### 一句话总结
> **哈希记录"我见过谁"，下一次就能 O(1) 知道"我要找的它在哪"。**

### 新手常见疑问（FAQ）

**Q1：为什么 `put` 写在 `if` 之后，而不是循环开头？**
A：如果先 `put`，遍历到 `nums[i]` 时表里就有它自己；当 `target == 2 * nums[i]` 时（如 `[3,3], target=6`），会用同一元素配自己，违反题意。「先查后写」是这个解法的灵魂。

**Q2：HashMap 不是 O(1) 平均、最坏 O(n) 吗？面试时怎么说？**
A：标准回答：「平均 O(1)，最坏 O(n)（哈希冲突极端情况下）。Java 8 起 HashMap 的桶在树化后最坏可降到 O(log n)。」面试按平均 O(n) 计即可。

**Q3：能不能不开 HashMap、O(1) 空间解？**
A：不能——只要要求 **返回原下标**，就必须记录"值→下标"的映射；`O(1)` 空间需要数组有序（用双指针），但本题没有有序假设。

**Q4：负数 / 溢出怎么办？**
A：本题 `nums[i]、target ∈ [-10^9, 10^9]`，`int` 加减不溢出。但若题面把范围放大到 `10^18`，要把 `target - nums[i]` 改成 `long`。

**Q5：返回顺序需要是 `[i, j], i<j` 吗？**
A：题目说「顺序不限」，但一次遍历的解法天然返回 `i < j`（更早入表的下标在前），通常无需额外排序。

### 面试官常见 follow-up
1. **"如果数组本身有序呢？"** → 改用双指针：左右各一指针向中间逼近，O(n) 时间 + O(1) 空间。这就是 **LC 167. Two Sum II**。
2. **"如果存在多对解，要全部返回呢？"** → 哈希表里存 `value → List<index>`，命中后枚举所有组合；要避免重复对要把 `i < j` 作为唯一性条件。
3. **"不许用 HashMap 怎么搞？"** → 排序 + 双指针（O(n log n)）；或用 `TreeMap`（红黑树，O(log n) 查询）。
4. **"数据是流式进来的（来一个数处理一个）呢？"** → 哈希方案天然支持在线，每来一个数 O(1) 处理并更新结果。
5. **"扩展到 3 数之和、4 数之和呢？"** → 排序后固定前 k-2 个元素，剩两个用双指针，时间 O(n^(k-1))。即 **LC 15 / LC 18**。
6. **"如果是 Two Sum 的子数组版（和为 K 的连续子数组）呢？"** → 前缀和 + 哈希：把 `prefixSum - K` 当 need 去 map 里查。即 **LC 560**。

### 自我提问
1. 为什么先查后写？→ 防止单元素被自己匹配（如 `target = 2 * nums[i]` 时）。
2. 数组有序时还需要哈希吗？→ 不需要，**双指针** 即可做到 O(n) 时间 + O(1) 空间。
3. 如果数据流式进来怎么办？→ 哈希表方案天然支持在线（每来一个数 O(1) 处理）。
